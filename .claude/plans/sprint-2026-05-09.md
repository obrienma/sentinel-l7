# ADR Closure Sprint

**Goal:** Turn documented-but-unwired ADR decisions into shipped code across Sentinel-L7, Synapse-L4, and EventHorizon. Time budget: 2–3 weekends.

**Why this sprint exists:** A gap audit across all three repos surfaced that several of the strongest architectural decisions (ADR-0018, ADR-0019, ADR-0020-adjacent work) are designed but not implemented. The READMEs present these systems as more complete than they are. Closing these loops is higher-leverage than starting a Greenfield project, because it demonstrates the rarest senior signal: turning your own ADRs into shipped code. After this sprint, the next Greenfield project (red-teaming node) has concrete failure modes to attack.

**Ground rules:**
- No new services. No new infrastructure. No scope creep into "while I'm in here…"
- Every change has an accompanying ADR update (status: Accepted → Implemented, with commit SHA).
- Every change has at least one test that would catch the regression if the wiring were removed.
- If a task is taking >2x its estimate, stop and write down why. The reason is portfolio-relevant.

## Task 1 — Wire up AI quality scoring (ADR-0019)

**Repo:** sentinel-l7
**Estimate:** 1 weekend (the anchor task)
**Why this is the most important task in the sprint:** ADR-0019 documents a real failure mode — Gemini returns structurally valid but poorly grounded responses, the pipeline `XACK`s, the dashboard stays green, and nobody knows the AI quietly degraded. You've designed the fix. It isn't wired up. An interviewer who reads ADR-0019 and then sees this implemented will weight that more heavily than any Greenfield service.

### Subtasks

**1.1 Schema migration**
- Add `quality_score` (tinyint, nullable, 0–4) to `compliance_events`.
- Add `quality_signals` (json, nullable) to store which of the 4 signals fired, for debuggability.
- Migration must be reversible.

**1.2 Quality scorer service**
- New class: `App\Services\Sentinel\Logic\QualityScorer`.
- Pure function: takes the Gemini response payload and the originating Axiom, returns `{score: int, signals: array}`.
- Lives in the protected `Logic` namespace per the existing Pest Arch test (no Http/Redis facades).
- The 4 signals from ADR-0019 — implement each as a private method returning bool:
  - `hasNarrativeOfSufficientLength` (configurable threshold, default 80 chars)
  - `hasPolicyCitation` (response references at least one retrieved policy chunk by id)
  - `hasDeterministicRiskLevel` (`risk_level` is one of low/medium/high/critical, not "unknown")
  - `hasGroundedJustification` (justification contains at least one entity from the input transaction — merchant, amount, account_id)

**1.3 Pipeline integration**
- In `ComplianceEngine::process` (or wherever the Gemini response is finalized), invoke the scorer before `XACK`.
- Persist `quality_score` and `quality_signals` to `compliance_events`.
- Do NOT block the `XACK` on a low score — the message is still processed. The score is observability, not flow control. (This is the key design point. Defend it in interviews: blocking on quality would create cascading retries on a probabilistic system. Better to surface and alert.)

**1.4 Alerting hook**
- Config flag `sentinel.quality.alert_threshold` (default: 1).
- When `score <= threshold`, dispatch a `LowQualityAiResponse` event (Laravel event, not a domain Axiom).
- Wire to a single listener that logs at `warning` level with full payload. Real alerting (Slack/PagerDuty) is out of scope — the hook exists, the integration doesn't.

**1.5 Dashboard surfacing**
- Add a "Low Quality AI Responses" tile to the Inertia dashboard showing count + last 10 over a 24h window.
- This is the smallest UI change that makes the new column visible. Resist the urge to build a full quality analytics view.

**1.6 ADR update**
- ADR-0019 status: Proposed → Implemented.
- Add an "Implementation Notes" section: which signals turned out to be noisy, which were silent, what the threshold was tuned to, links to relevant commits.

### Definition of done
- A Pest test that constructs a Gemini response with each of the 4 failure modes and asserts the score drops accordingly.
- A Pest test that asserts the listener fires when score ≤ threshold.
- ADR-0019 updated with commit SHAs.
- The dashboard tile renders without errors when there are zero low-quality events (this is the boring bug that will bite you).

### Out of scope (write these down so you don't drift)
- Tuning thresholds against real traffic — that's continuous, not a sprint task.
- Auto-remediation when a low score is detected.
- Per-tenant thresholds (still gated by ADR-0020 multi-tenancy deferral).

## Task 2 — Stamp `domain` on Axioms and filter policy retrieval (ADR-0018)

**Repo:** synapse-l4 + sentinel-l7
**Estimate:** Half weekend
**Why this matters:** ADR-0018 documents that as the policy corpus grows, a high-scoring GDPR chunk can outrank lower-scoring AML chunks for an AML event, and Gemini will reason confidently over the wrong grounding doc. The mitigation (metadata filter on `domain`) exists in the Vector layer; the upstream pipeline doesn't stamp `domain`, so the unfiltered path is what runs in production today.

### Subtasks

**2.1 Synapse-L4: extend the Axiom schema**
- Add `domain` field to the Axiom Pydantic model — Literal type: `aml | hipaa | gdpr | pci_dss | api_governance | unknown`.
- Default to `unknown` if the extractor can't classify.
- Update the Instructor extraction prompt to request domain classification as part of the structured output (this is essentially free given Instructor — you're getting another field, not another LLM call).

**2.2 Synapse-L4: tests**
- Pytest cases for each domain — given a representative raw telemetry payload, assert the extracted Axiom carries the expected domain.
- One adversarial test: a payload that could plausibly be AML or GDPR — assert the result is deterministic across runs (low temperature, structured output should stabilize this; if it doesn't, log the finding in ADR-0018 update).

**2.3 Sentinel-L7: consume the domain field**
- `AxiomProcessorService` reads `domain` from the inbound Axiom payload.
- Pass `domain` into the policy RAG query as a metadata filter on the `policies` namespace.
- If `domain == 'unknown'`, log a warning and fall through to the unfiltered path (preserves backward compat during rollout).

**2.4 Sentinel-L7: ingestion side**
- Update `php artisan sentinel:ingest` to write `domain` metadata onto each policy chunk during upsert.
- One-time backfill script for the existing policies in the namespace.

**2.5 ADR update**
- ADR-0018 status update.
- Add a section quantifying retrieval precision before/after on a small held-out set if you have one. If you don't, write down that you don't, and that this is a measurement gap (which becomes a red-teaming node task later).

### Definition of done
- An integration test (or recorded fixture) showing that an AML transaction now retrieves only AML-tagged policy chunks.
- Domain-stamped Axioms visible in the Sentinel logs.
- ADR updated.

### Out of scope
- A confusion matrix across all domains. That's eval-plane work.
- Reclassifying historical Axioms.

## Task 3 — Extend XCLAIM recovery to the Axioms stream

**Repo:** sentinel-l7
**Estimate:** 2–4 hours
**Why this matters:** The reclaimer pattern is one of the headline patterns in the Sentinel README. It's only wired to the `transactions` stream. The `synapse:axioms` stream — which carries the output of the entire Synapse pipeline — has no recovery, so an Axioms worker crash mid-process drops messages. This is the cheapest possible "your README is more complete than your code" gap to close.

### Subtasks
- Parameterize the existing `Reclaimer` daemon to take a stream key + consumer group as constructor args (if not already).
- Register a second instance against `synapse:axioms` in the service provider / `composer dev-full` script.
- Update `Procfile` and `render.yaml` to run both reclaimers.
- Update the system diagram in the README to show recovery on both streams (this is the visible portfolio change).

### Definition of done
- Kill -9 an Axioms worker mid-process in a local dev run; observe XCLAIM picks up the pending message after `min-idle-time` elapses.
- README diagram updated.

### Out of scope
- Generalizing reclaimer config to a registry. Premature. You have two streams.

## Task 4 — EventHorizon: handle oplog overrun (ADR-0011)

**Repo:** EventHorizon
**Estimate:** 2–3 hours
**Why this matters:** The inline TODO in `src/observation/changeStream.ts` and ADR-0011 both flag this. If MongoDB rolls past the resume token during a long server outage, the retry loop stalls at `RETRY_MAX_MS` and the observation plane is silently dead. Cheap to fix, removes a "documented bug" from the repo.

### Subtasks
- In the change stream error handler, inspect the MongoDB error code.
- On code 286 (`ChangeStreamHistoryLost`), emit a structured log line and restart the change stream tokenless.
- Document the trade-off: tokenless restart means events between the lost-token moment and the restart are not replayed to the live feed. Acceptable per ADR-0008 (observation plane is best-effort).
- Add a unit test using a mocked Mongo error.

### Definition of done
- Inline TODO removed.
- Test asserts tokenless fallback on code 286.
- ADR-0011 status updated.

## Task 5 — EventHorizon: `safeParse` in the worker

**Repo:** EventHorizon
**Estimate:** 1 hour
**Why this matters:** Inline TODO in `src/processing/worker.ts`. Currently a single malformed message throws and crashes the consumer. With `safeParse`, bad messages route to DLQ and the consumer keeps eating the queue. This is the smallest possible "production hardening" change.

### Subtasks
- Replace `EventSchema.parse` with `EventSchema.safeParse` in the worker.
- On parse failure, `nack` to DLQ with the validation errors attached as a header.
- Add a vitest case: feed a malformed message, assert no throw, assert DLQ routing.

### Definition of done
- The TODO comment is gone, replaced by a brief inline note explaining the DLQ contract.

## What you should NOT do during this sprint

- Build the federated auth plane. (Even though Sentinel's MCP endpoint has no OAuth — that's roadmap, not sprint.)
- Start the eval plane or the red-teaming node. (Roadmap.)
- "Quickly" add multi-tenancy because the quality-score work made you think about per-tenant thresholds. ADR-0020 still holds.
- Add new ADRs unless you make a genuinely new architectural decision. Updating existing ADRs is the goal.
- Refactor for the sake of it. The Service/Manager pattern in Sentinel is fine as-is for this scope.
