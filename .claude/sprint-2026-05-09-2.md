# ADR Closure Sprint (Revised)

**Goal:** Turn documented-but-unwired ADR decisions into shipped code across Sentinel-L7, Synapse-L4, and EventHorizon. Time budget: 1 weekend.

**Why this sprint exists:** A gap audit across all three repos surfaced that several architectural decisions documented in ADRs were not yet reflected in code. A subsequent review against the actual repo state showed the gap is smaller than the audit suggested ‚Äî much of the work is already done or deliberately scoped narrower than the audit assumed. This revised sprint reflects what's actually left.

**Revision note:** An earlier version of this sprint included a Task 1 to expand ADR-0019 (AI quality scoring) from log-only to a persisted column with event dispatch and dashboard surfacing. That task has been removed because it would reverse a decision the ADR explicitly documents. See "Considered and rejected" at the bottom of this file for the reasoning, so future planning sessions don't re-propose it without a real trigger.

**Ground rules:**
- No new services. No new infrastructure. No scope creep into "while I'm in here‚Ä¶"
- Every change has an accompanying ADR update (status update with commit SHA, or a new ADR if the decision is genuinely new).
- Every change has at least one test that would catch the regression if the wiring were removed.
- If a task is taking >2x its estimate, stop and write down why. The reason is portfolio-relevant.
- New code in Sentinel goes under `app/Services/Compliance/` to match the actual repo layout (not `app/Services/Sentinel/Logic/` as the README's class diagram suggests).

## Task 1 ‚Äî Stamp `domain` on Axioms in `AxiomProcessorService` (ADR-0018 completion)

**Repo:** sentinel-l7
**Estimate:** 1‚Äì2 hours
**Why this is the only ADR-0018 task remaining:** The driver-side domain filtering, the ingest-side tagging, and the retrieval-quality logging are already implemented and the ADR is Accepted. The one piece left is the upstream stamping in `AxiomProcessorService` so that domain flows through the pipeline rather than relying on the unfiltered fall-through path.

### Subtasks

**1.1 Read the existing domain contract**
- Confirm what shape `domain` takes when the driver consumes it (Literal values, optional vs. required, default behavior).
- Match that contract exactly. Don't redesign the field.

**1.2 Stamp `domain` in `AxiomProcessorService`**
- Wherever the service composes the Axiom payload that the driver receives, populate `domain`.
- Source of truth: whatever Synapse-L4 already includes in its outbound Axioms (Synapse's domain stamping work was completed earlier ‚Äî confirm the field is on the wire).
- If Synapse isn't yet emitting `domain`, that's a Synapse-side task ‚Äî see Task 1.5.

**1.3 Fall-through behavior**
- If the inbound Axiom lacks `domain`, log at `warning` level and pass `unknown` (or whatever the driver-side default already expects) so the unfiltered retrieval path runs. This preserves backward compatibility while making the absence visible.

**1.4 Test**
- Pest test: feed an Axiom with a known domain through `AxiomProcessorService`, assert the driver receives the domain in the expected field.
- Pest test: feed an Axiom missing domain, assert the warning log fires and the request still completes.

**1.5 Synapse-L4 confirmation (separate repo, may be a no-op)**
- If Synapse already stamps `domain` on its outbound Axioms, this is a confirmation step only.
- If it doesn't, add it: extend the Axiom Pydantic model with `domain: Literal[...]`, update the Instructor extraction prompt to classify, default to `unknown` on classifier uncertainty.
- Document which case turned out to be true in the ADR-0018 update.

**1.6 ADR update**
- ADR-0018: add an "Implementation Notes" section listing the commits that completed each layer (driver, ingest, retrieval logging, AxiomProcessorService stamping).
- If a wrong-domain retrieval rate has been measured anywhere in the existing logging, link it. If not, note that quantifying retrieval precision is a Phase 2 (red-teaming) measurement task.

### Definition of done
- Axioms flowing through `AxiomProcessorService` carry `domain` end-to-end.
- The fall-through warning fires when expected.
- ADR-0018 reflects the full implementation chain.

### Out of scope
- Tuning the domain classifier in Synapse against a held-out set. Phase 2 work.
- Adding new domain values beyond what's already in the Literal type.

## Task 2 ‚Äî Extend XCLAIM recovery to the Axioms stream

**Repo:** sentinel-l7
**Estimate:** 2‚Äì4 hours
**Why this matters:** The reclaimer pattern is one of the headline patterns in the Sentinel README. It's only wired to the `transactions` stream. The `synapse:axioms` stream ‚Äî which carries the output of the entire Synapse pipeline ‚Äî has no recovery, so an Axioms worker crash mid-process drops messages. This is the cheapest "your README is more complete than your code" gap to close.

### Subtasks
- Parameterize the existing `Reclaimer` daemon to take a stream key + consumer group as constructor args (if not already).
- Register a second instance against `synapse:axioms` in the service provider / `composer dev-full` script.
- Update `Procfile` and `render.yaml` to run both reclaimers.
- Update the system diagram in the README to show recovery on both streams (this is the visible portfolio change).

### Definition of done
- Kill -9 an Axioms worker mid-process in a local dev run; observe XCLAIM picks up the pending message after `min-idle-time` elapses.
- README diagram updated.

### Out of scope
- Generalizing reclaimer config to a registry. Premature. You have two streams.

## Task 3 ‚Äî EventHorizon: handle oplog overrun (ADR-0011)

**Repo:** EventHorizon
**Estimate:** 2‚Äì3 hours
**Why this matters:** The inline TODO in `src/observation/changeStream.ts` and ADR-0011 both flag this. If MongoDB rolls past the resume token during a long server outage, the retry loop stalls at `RETRY_MAX_MS` and the observation plane is silently dead. Cheap to fix, removes a documented bug.

### Subtasks
- In the change stream error handler, inspect the MongoDB error code.
- On code 286 (`ChangeStreamHistoryLost`), emit a structured log line and restart the change stream tokenless.
- Document the trade-off: tokenless restart means events between the lost-token moment and the restart are not replayed to the live feed. Acceptable per ADR-0008 (observation plane is best-effort).
- Add a unit test using a mocked Mongo error.

### Definition of done
- Inline TODO removed.
- Test asserts tokenless fallback on code 286.
- ADR-0011 status updated.

## Task 4 ‚Äî EventHorizon: `safeParse` in the worker

**Repo:** EventHorizon
**Estimate:** 1 hour
**Why this matters:** Inline TODO in `src/processing/worker.ts`. Currently a single malformed message throws and crashes the consumer. With `safeParse`, bad messages route to DLQ and the consumer keeps eating the queue. Smallest possible production-hardening change.

### Subtasks
- Replace `EventSchema.parse` with `EventSchema.safeParse` in the worker.
- On parse failure, `nack` to DLQ with the validation errors attached as a header.
- Add a vitest case: feed a malformed message, assert no throw, assert DLQ routing.

### Definition of done
- TODO comment is gone, replaced by a brief inline note explaining the DLQ contract.

## Considered and rejected

This section exists so future planning sessions don't re-propose work that has already been deliberately scoped out. If a future trigger justifies revisiting any of these, write a new ADR ‚Äî don't silently expand scope.

### ADR-0019 expansion: persisted `quality_score` column + event dispatch + dashboard tile

**Why it was proposed:** A gap-audit summary described ADR-0019 as "Proposed, designed but not wired." On that basis, an earlier version of this sprint included a task to add a `quality_score` column to `compliance_events`, dispatch a `LowQualityAiResponse` event on low scores, and surface the data in a dashboard tile.

**Why it was rejected:**
1. ADR-0019 is **Accepted**, not Proposed. The current implementation ‚Äî `GeminiDriver::logResponseQuality()` (`app/Services/Compliance/GeminiDriver.php:143`, shipped in commit `12998b7`) ‚Äî is the deliberate scope of the ADR. It's not a partial implementation pending the bigger version; it's the chosen design.
2. The persisted-column / event-dispatch / dashboard path is explicitly listed under **Alternatives considered** in the ADR and rejected at the time it was written. Re-implementing it now without a new trigger would reverse a documented decision.
3. The pattern is consistent with the rest of the project's ADR philosophy ("wait until it hurts" ‚Äî see ADR-0017 on prompt governance, ADR-0020 on multi-tenancy). Log-only is the right scope until something downstream needs queryable quality history.

**What would justify revisiting this:**
- Phase 2 (red-teaming node) needing to query historical quality scores to measure attack success rates over time. Logs are not queryable enough for that workload.
- A specific incident where the log-only signal was insufficient to diagnose a quality regression after the fact.
- Multiple consumers wanting to react to low quality scores (alerting, routing, retry logic) ‚Äî at which point the event-dispatch path earns its keep.

**If revisited:** write a new ADR (e.g., ADR-0019a or a superseding ADR per the project's convention) that records the trigger and explicitly supersedes the relevant section of ADR-0019. Don't edit ADR-0019 in place to flip the decision; the historical record matters.
